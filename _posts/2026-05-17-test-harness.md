# 하네스의 개념

## 1. 테스트 하네스란?

테스트 하네스(Test Harness)는 테스트 대상(SUT)을 정해진 환경 안에서 실행하고, 입력 주입, 의존성 제어, 결과 수집, 검증을 반복 가능하게 만들어주는 테스트용 실행 틀이다.

쉽게 말하면, 내가 검증하려는 코드나 모델을 특정한 테스트 장치 안에 끼워 넣고, 원하는 입력을 넣은 뒤 결과가 기대한 대로 나오는지 확인할 수 있게 해주는 구조이다.

여기서 중요한 점은 하네스가 항상 모든 외부 환경을 완전히 제거한다는 뜻은 아니라는 것이다. 테스트 목적에 따라 외부 DB, 네트워크 API, 환경 변수, 파일 시스템 같은 의존성을 실제로 연결할 수도 있고, 필요한 경우 스텁이나 모크로 대체할 수도 있다.

## 2. 하네스가 해결하려는 근본적인 문제

어떤 기능을 개발했다고 가정해보자. 예를 들어 "사용자의 장바구니 데이터를 읽어와서 총 결제 금액을 계산하는 함수"가 있다고 하자.

이 함수를 실제 서비스 환경 그대로 테스트하려면 다음과 같은 과정이 필요할 수 있다.

1. 실제 DB에 접속해서 장바구니 데이터를 가져와야 함
2. 외부 PG API와 통신해서 환율이나 수수료 정보를 가져와야 함
3. 웹 서버나 백엔드 서비스를 함께 실행해야 함
4. 테스트가 실패했을 때 원인이 계산 로직인지, DB인지, 외부 API인지 구분해야 함

이런 방식은 테스트가 느리고 불안정해지기 쉽다. 외부 API가 잠시 느려지거나 DB 상태가 바뀌어도 테스트 결과가 흔들릴 수 있기 때문이다.

테스트 하네스는 이 문제를 해결하기 위해 테스트 대상 주변의 실행 환경을 통제한다.

- 테스트 입력을 미리 정해진 값으로 고정함
- 필요한 경우 DB, 네트워크, 외부 API 같은 의존성을 스텁/모크로 대체함
- 테스트 대상이 반환한 결과를 자동으로 수집함
- 기대값과 실제 결과를 비교함
- 성공/실패 로그나 리포트를 남김

즉 하네스의 목적은 테스트 대상을 더 안정적이고 반복 가능한 방식으로 검증하는 것이다.

## 3. 테스트 하네스의 대표 구성 요소

테스트 하네스는 상황에 따라 구성이 달라질 수 있지만, 대표적으로 다음 요소들을 포함할 수 있다.

![테스트 하네스 흐름](/assets/img/posts/ai/harness/test-harness-flow.png)

### 3-1. 테스트 드라이버(Test Driver)

테스트 드라이버는 테스트 대상의 바깥에서 테스트를 시작하고 흐름을 제어하는 역할을 한다.

예를 들어 테스트 케이스 파일(JSON, CSV 등)을 읽어와서 테스트 대상에게 입력값을 전달하고, 실행을 트리거한다. 실행이 끝나면 결과를 받아서 검증 로직이나 리포터로 넘긴다.

### 3-2. 테스트 대상(SUT: System Under Test)

SUT는 현재 검증하려는 대상이다.

일반적인 소프트웨어 테스트에서는 함수, 클래스, 모듈, API, 서비스, 전체 프로그램이 SUT가 될 수 있다. AI 평가 문맥에서는 LLM, 에이전트, RAG 시스템, 추천 모델 같은 모델 또는 시스템도 SUT가 될 수 있다.

### 3-3. 테스트 스텁 / 모크(Test Stub / Mock)

스텁이나 모크는 테스트 대상이 실행 중에 의존하는 외부 요소를 가짜로 대체하는 장치이다.

예를 들어 대상 코드가 DB, 네트워크 API, 하드웨어 센서, 파일 시스템을 호출한다면, 하네스는 이를 실제 환경 대신 미리 정해진 응답을 반환하는 가짜 객체로 바꿀 수 있다.

- 스텁: 미리 정해진 값을 반환하는 단순한 대체 구현
- 모크: 호출 여부, 호출 횟수, 전달된 인자까지 검증할 수 있는 대체 객체

스텁/모크는 항상 필수는 아니지만, 외부 의존성 때문에 테스트가 느려지거나 불안정해질 때 자주 사용된다.

### 3-4. 검증기 / 리포터

검증기는 실제 결과가 기대 결과와 일치하는지 판단한다.

리포터는 테스트 결과를 사람이 읽을 수 있는 형태나 기계가 처리할 수 있는 형태로 저장한다. 예를 들어 콘솔 출력, HTML 리포트, JSON 로그, 점수표 등이 여기에 해당한다.

## 4. 단순 테스트 코드와 테스트 하네스의 차이

테스트 코드는 개별적인 검증 명제에 가깝고, 테스트 하네스는 그 명제들이 안정적으로 실행될 수 있도록 받쳐주는 실행 인프라에 가깝다.

- 테스트 코드: "A를 넣으면 B가 나와야 한다"라는 검증 로직
- 테스트 하네스: 테스트 코드를 실행하기 위해 입력을 준비하고, 환경을 통제하고, 타임아웃을 체크하고, 결과를 수집하고, 리포트를 생성하는 구조

예를 들어 단순 테스트 코드는 다음과 같은 한 줄의 검증일 수 있다.

```python
assert calculate_total(cart) == 30000
```

반면 테스트 하네스는 이 검증이 안정적으로 실행되도록 테스트 데이터를 준비하고, 외부 API를 모킹하고, 실행 시간을 제한하고, 실패 로그를 남기는 전체 구조를 의미한다.

## 5. 하네스 드라이버

하네스 드라이버(Test Driver)는 테스트 하네스 안에서 테스트 대상(SUT)을 실제로 실행시키는 역할을 담당한다.

테스트 대상은 보통 자기 혼자 테스트 시나리오를 시작하지 않는다. 따라서 드라이버가 미리 준비된 입력값, 테스트 케이스, 설정값을 읽어와서 테스트 대상에게 전달하고 실행을 트리거한다. 실행이 끝나면 드라이버는 결과를 받아 기대값과 비교하거나, 별도의 채점기/리포터로 넘긴다.

즉 드라이버는 하네스 내부에서 **테스트 흐름을 위에서부터 제어하는 실행 주체**라고 볼 수 있다.

### 5-1. 일반 소프트웨어 테스트에서의 드라이버

예를 들어 장바구니 총액 계산 함수를 테스트한다고 하면 드라이버는 다음과 같은 일을 한다.

1. 테스트용 장바구니 데이터를 준비함
2. 총액 계산 함수에 데이터를 전달함
3. 함수가 반환한 총액을 받음
4. 기대한 총액과 실제 총액을 비교함
5. 성공/실패 결과를 기록함

이때 장바구니 총액 계산 함수가 테스트 대상(SUT)이고, 그 함수를 호출해서 입력을 넣고 결과를 확인하는 쪽이 드라이버이다.

### 5-2. AI 평가 하네스에서의 드라이버

AI 평가 하네스에서는 드라이버가 평가 데이터셋을 순회하면서 각 샘플을 모델이 이해할 수 있는 입력 형태로 바꿔준다.

예를 들어 LLM 평가에서는 드라이버가 다음과 같은 흐름을 제어한다.

1. 평가 데이터셋에서 문제 하나를 가져옴
2. 문제를 프롬프트 템플릿에 끼워 넣음
3. 모델에게 프롬프트를 입력함
4. 모델의 응답을 수집함
5. 응답을 채점기나 메트릭 계산기로 넘김
6. 최종 점수와 로그를 저장함

이 관점에서 보면 AI 모델은 테스트 대상(SUT)이고, 평가 데이터를 넣어 모델을 실행시키는 쪽이 드라이버이다.

### 5-3. 드라이버와 스텁의 차이

드라이버와 스텁은 둘 다 테스트 하네스 안에서 테스트 대상을 도와주는 장치이지만 위치와 방향이 다르다.

- 드라이버: 테스트 대상의 바깥/위쪽에서 테스트 대상을 호출하는 역할
- 스텁/모크: 테스트 대상의 안쪽/아래쪽에서 외부 의존성을 대신하는 역할

즉 드라이버는 테스트 대상을 실행시키는 쪽이고, 스텁/모크는 테스트 대상이 실행 중에 의존하는 외부 요소를 대신하는 쪽이다.

## 6. 가드레일(Guardrail)과 하네스의 관계

가드레일(Guardrail)은 AI 시스템이 위험하거나 원하지 않는 행동을 하지 않도록 입력, 출력, 실행 과정을 제한하거나 검사하는 안전 장치이다.

하네스가 "테스트 대상이 정해진 조건에서 잘 실행되는지 평가하는 틀"이라면, 가드레일은 "테스트 대상이 허용된 범위를 벗어나지 않도록 막는 제어 장치"에 가깝다.

특히 AI 모델이나 에이전트는 단순 함수보다 실행 결과가 예측하기 어렵다. 같은 입력이라도 모델 설정, 프롬프트, 도구 사용 가능 여부에 따라 다른 응답이나 행동이 나올 수 있다. 그래서 AI 평가 하네스에서는 모델을 실행하고 채점하는 것뿐 아니라, 위험한 입력과 출력, 도구 사용, 시간 초과 같은 문제를 제어하는 가드레일이 중요해진다.

![NVIDIA NeMo Guardrails high-level flow](https://docs.nvidia.com/nemo/guardrails/0.19.0/_images/programmable_guardrails_flow.png)

출처: [NVIDIA NeMo Guardrails - Guardrails Process](https://docs.nvidia.com/nemo/guardrails/0.19.0/user-guides/guardrails-process.html)

### 6-1. 입력 가드레일

입력 가드레일은 모델에게 전달되기 전의 입력을 검사한다.

예를 들어 다음과 같은 입력을 차단하거나 별도 처리할 수 있다.

- 개인정보가 포함된 입력
- 시스템 프롬프트를 탈취하려는 프롬프트 인젝션
- 금지된 행동을 유도하는 악성 요청
- 평가 목적과 관계없는 비정상적으로 긴 입력

입력 가드레일은 모델이 위험한 요청을 처리하기 전에 앞단에서 걸러주는 역할을 한다.

### 6-2. 출력 가드레일

출력 가드레일은 모델이 생성한 응답을 사용자나 다음 시스템으로 넘기기 전에 검사한다.

예를 들어 다음과 같은 내용을 탐지할 수 있다.

- 개인정보 또는 민감정보 노출
- 유해하거나 정책을 위반하는 응답
- 허위 사실 가능성이 높은 응답
- 정해진 JSON, 코드, 표 형식을 깨뜨린 응답

출력 가드레일은 모델의 답변이 그대로 외부에 나가기 전에 마지막 안전 필터 역할을 한다.

### 6-3. 실행 가드레일

실행 가드레일은 에이전트가 도구를 사용하거나 외부 시스템에 영향을 줄 때 필요한 제한 장치이다.

예를 들어 AI 에이전트가 파일 시스템, 터미널, 브라우저, API, 결제 시스템을 사용할 수 있다면 다음과 같은 제어가 필요하다.

- 파일 삭제나 덮어쓰기 전에 승인 받기
- 외부 API 호출 횟수 제한하기
- 네트워크 접근 범위 제한하기
- 시스템 명령 실행 권한 제한하기
- 무한 루프나 과도한 도구 호출을 타임아웃으로 중단하기

즉 실행 가드레일은 AI가 실제 환경에 영향을 주는 행동을 할 때 사고를 막는 안전 울타리이다.

### 6-4. 평가 가드레일

평가 가드레일은 모델 평가 과정 자체가 공정하고 안정적으로 진행되도록 돕는다.

예를 들어 다음과 같은 제어가 가능하다.

- 모든 모델에 동일한 프롬프트와 조건 적용
- 응답 길이 제한
- 평가 시간 제한
- 실패한 샘플 로그 저장
- 모델이 평가 정답을 직접 참조하지 못하도록 데이터 누수 차단

이런 장치가 있어야 여러 모델을 비교할 때 평가 결과를 신뢰할 수 있다.

### 6-5. 하네스와 가드레일의 차이

하네스와 가드레일은 함께 쓰일 수 있지만 초점이 다르다.

- 하네스: 모델이나 코드를 반복 가능하게 실행하고 평가하는 틀
- 가드레일: 실행 과정이 위험하거나 원하지 않는 방향으로 벗어나지 않게 막는 안전 장치

따라서 AI 평가 하네스는 내부에 여러 가드레일을 포함할 수 있다. 이때 하네스는 전체 평가 흐름을 제어하고, 가드레일은 그 흐름 안에서 입력, 출력, 도구 사용, 평가 조건을 제한한다.

## 7. AI 하네스에서 함께 쓰이는 운영 구성 요소

AI 평가 하네스는 단순히 모델을 호출하고 점수를 내는 도구에서 끝나지 않는다. 실제 AI 시스템을 평가하거나 운영하려면 비용, 속도, 코드 안전성, 로그 추적 같은 문제도 함께 다뤄야 한다.

이때 자주 함께 등장하는 개념이 시맨틱 캐시, 코드 검증기, 옵저버빌리티이다.

### 7-1. 시맨틱 캐시(Semantic Cache)

시맨틱 캐시는 입력 문장이 완전히 같지 않아도 의미가 비슷하면 이전에 생성한 응답이나 계산 결과를 재사용하는 캐시이다.

일반 캐시는 보통 문자열이 정확히 같을 때만 동작한다. 예를 들어 "배송 상태 알려줘"와 "내 택배 어디쯤 왔어?"는 문자열은 다르지만 의미는 비슷하다. 시맨틱 캐시는 이런 문장들을 임베딩 벡터로 변환한 뒤, 벡터 공간에서 가까운 요청을 찾아 이전 결과를 재사용할 수 있게 한다.

AI 하네스에서 시맨틱 캐시는 다음과 같은 역할을 할 수 있다.

- 반복되는 모델 호출 비용 줄이기
- 비슷한 평가 샘플에 대한 응답 재사용하기
- RAG 검색 결과나 임베딩 결과를 재사용하기
- 같은 의미의 요청이 들어왔을 때 응답 일관성 확인하기

다만 시맨틱 캐시는 항상 정답을 보장하지 않는다. 의미가 비슷해 보여도 실제로는 다른 답이 필요한 경우가 있기 때문이다. 따라서 평가 하네스에서는 캐시 적중 기준, 유사도 임계값, 캐시 사용 여부를 로그로 남기는 것이 중요하다.

### 7-2. 코드 검증기(Code Verifier)

코드 검증기는 AI가 생성한 코드가 실제로 실행 가능한지, 테스트를 통과하는지, 위험한 동작을 포함하지 않는지 확인하는 장치이다.

LLM이나 코딩 에이전트는 코드처럼 보이는 텍스트를 생성할 수 있지만, 그 코드가 실제로 맞는지는 별도로 검증해야 한다. 코드 검증기는 생성된 코드를 샌드박스에서 실행하거나 정적 분석, 타입 체크, 단위 테스트, 보안 규칙 검사를 통해 결과를 확인한다.

AI 하네스에서 코드 검증기는 다음과 같은 일을 할 수 있다.

- 생성된 코드가 문법적으로 유효한지 검사함
- 타입 체크나 린터를 실행함
- 준비된 테스트 케이스를 실행함
- 파일 삭제, 네트워크 호출, 시스템 명령 같은 위험한 동작을 탐지함
- 실행 결과, 에러 로그, 테스트 통과 여부를 리포트로 남김

즉 코드 검증기는 모델의 답변을 단순 텍스트로 평가하지 않고, 실제 실행 결과를 기준으로 검증하는 장치이다.

### 7-3. 옵저버빌리티(Observability)

옵저버빌리티는 AI 시스템 내부에서 무슨 일이 일어났는지 추적하고 분석할 수 있게 만드는 관측 가능성이다.

![OpenTelemetry GenAI trace span tree](https://opentelemetry.io/blog/2026/genai-observability/dashboard-spans.png)

출처: [OpenTelemetry - Inside the LLM Call: GenAI Observability](https://opentelemetry.io/blog/2026/genai-observability/)

일반적인 로그가 "무슨 일이 발생했는지"를 기록한다면, 옵저버빌리티는 그 로그, 메트릭, 트레이스를 바탕으로 "왜 그런 일이 발생했는지"까지 추적할 수 있게 해준다.

AI 하네스에서 옵저버빌리티는 다음 정보를 기록할 수 있다.

- 어떤 프롬프트가 모델에 들어갔는지
- 어떤 모델과 파라미터를 사용했는지
- 응답 생성에 걸린 시간
- 토큰 사용량과 비용
- 도구 호출 순서
- RAG에서 검색된 문서
- 가드레일에 의해 차단된 입력/출력
- 평가 점수와 실패 사례

옵저버빌리티가 없으면 평가 결과가 나쁘게 나왔을 때 원인을 찾기 어렵다. 모델 자체가 문제인지, 프롬프트가 문제인지, 검색 문서가 문제인지, 도구 호출이 실패했는지 구분하기 힘들기 때문이다.

### 7-4. 세 개념의 관계

시맨틱 캐시, 코드 검증기, 옵저버빌리티는 각각 다른 문제를 해결한다.

- 시맨틱 캐시: 비슷한 요청의 결과를 재사용하여 비용과 시간을 줄임
- 코드 검증기: AI가 만든 코드나 실행 결과가 실제로 안전하고 올바른지 확인함
- 옵저버빌리티: 하네스 내부의 실행 흐름, 비용, 실패 원인을 추적함

이 세 가지는 하네스의 필수 구성 요소라기보다는, AI 하네스를 실제 시스템 수준으로 운영하기 위해 자주 붙는 보조 인프라라고 보는 것이 정확하다.

## 8. AI 공부 관점에서의 정리

AI 문맥에서 하네스는 보통 모델이나 에이전트를 평가하기 위한 실행 시스템이라는 의미로 자주 쓰인다.

예를 들어 LLM 평가 하네스는 다음과 같은 일을 한다.

1. 평가 데이터셋을 불러옴
2. 문제를 프롬프트로 변환함
3. 모델을 호출함
4. 모델 응답을 수집함
5. 정답과 비교하거나 별도 기준으로 채점함
6. 모델별 점수, 로그, 실패 사례를 저장함

따라서 AI 평가 하네스는 단순히 모델에게 질문을 던지는 코드가 아니라, 여러 모델을 같은 조건에서 반복적으로 평가하고 비교할 수 있게 해주는 평가 인프라라고 볼 수 있다.

## 핵심 요약

- 테스트 하네스는 테스트 대상의 실행 환경, 입력, 의존성, 결과 검증을 통제하는 테스트용 실행 틀이다.
- 테스트 코드는 개별 검증 로직이고, 테스트 하네스는 그 검증이 안정적으로 반복 실행되도록 돕는 구조이다.
- 드라이버는 테스트 대상을 호출하고 실행 흐름을 제어한다.
- 스텁/모크는 테스트 대상이 의존하는 외부 요소를 대신한다.
- AI 평가 하네스에서는 모델, 에이전트, RAG 시스템 등이 SUT가 될 수 있다.
- 가드레일은 AI 시스템의 입력, 출력, 실행 과정을 제한하여 위험하거나 원하지 않는 행동을 막는다.
- 시맨틱 캐시는 의미가 비슷한 요청의 결과를 재사용해 비용과 시간을 줄인다.
- 코드 검증기는 AI가 생성한 코드가 실제로 실행 가능하고 안전한지 확인한다.
- 옵저버빌리티는 프롬프트, 응답, 도구 호출, 비용, 실패 원인을 추적할 수 있게 한다.
